# 调试能力训练指南

这份指南不是教你“点哪个按钮”，而是教你形成真正的排错思路。

## 一、先建立正确观念

很多初学者把调试理解成：
- 程序错了
- 点几下断点
- 看看哪里红了

这不够。

真正的调试应该是：
1. 先看现象
2. 再猜原因
3. 然后设计验证步骤
4. 最后确认根因并修复

## 二、你每次排错都按这个顺序来

### 第一步：先看“现象”

你要先说清楚：
- 程序是编译不过
- 还是运行时报错
- 还是结果不对
- 还是卡住不动

不同现象，排查顺序完全不同。

### 第二步：找“最后一个正常点”

比如：
- 输入是正常的吗
- 变量刚赋值时正常吗
- 进入函数前正常吗
- 循环前正常吗

不要一上来就在崩溃那一行死盯，要先找到“从哪开始变不对”。

### 第三步：打断点

最常打断点的位置：
- 变量刚赋值后
- 函数入口
- 循环内部
- 条件判断前
- 报错前一行

### 第四步：单步执行

你要学会区分：
- 逐过程
- 逐语句
- 跳出函数

初学阶段最重要的是：
- 变量值有没有变
- 变成了什么
- 为什么在这里变了

### 第五步：修复后再验证

不要改完就以为结束了。

你要再检查：
- 原来的报错还在不在
- 新的边界情况会不会再错
- 这次修复是不是只是“碰巧好了”

## 三、你现在最需要练的 5 类 bug

### 1. 下标越界

典型现象：
- C++ 结果异常
- C# 直接抛异常

调试重点：
- 数组长度是多少
- 当前下标是多少
- 合法范围是什么

### 2. 空对象 / 空指针

典型现象：
- C# 空引用异常
- C++ 崩溃或访问非法内存

调试重点：
- 这个对象有没有初始化
- 是谁把它设为空了
- 在哪一步开始失效

### 3. 条件判断写错

典型现象：
- 分支一直不进
- 或者总进错分支

调试重点：
- 判断前变量实际值是多少
- 比较符号是不是写反了
- 逻辑与 / 逻辑或是不是用错了

### 4. 循环问题

典型现象：
- 死循环
- 少执行一次
- 多执行一次

调试重点：
- 循环变量初始值
- 每次变化量
- 退出条件

### 5. 函数参数传递问题

典型现象：
- 改了变量却没生效
- 返回值不对

调试重点：
- 是值传递还是引用传递
- 函数里拿到的值到底是多少
- 调用前后变量有没有变化

## 四、你练调试时要强迫自己回答的问题

每次打断点时，你都问自己：

1. 我预期这个变量现在应该是多少？
2. 实际它是多少？
3. 如果不一样，是从哪一步开始偏掉的？
4. 我下一步要验证哪一个猜想？

只要你坚持这样想，调试能力会涨得很快。

## 五、VS Code 里最常用的几个调试动作

### 对 C++
- 在代码左侧点红点：加断点
- `F5`：开始调试
- `F10`：逐过程
- `F11`：单步进入
- 监视窗口：看变量值

### 对 C#
- `F5`：启动调试
- 在异常前打断点
- 看“局部变量”和“调用堆栈”
- 特别关注异常信息里的出错行

## 六、你接下来怎么练最有效

建议你按这个顺序训练：

1. 先学一节
2. 运行示例
3. 故意改错
4. 自己调试回来
5. 把“错因”和“定位过程”写在 `Notes.md`

## 七、我对你的建议

你现在不要追求“一次写对所有代码”。

你真正要培养的是：
- 看到错不慌
- 知道先查哪里
- 会用断点和变量观察
- 能把 bug 定位清楚

这套能力，才是以后做项目、做面试题、进公司最值钱的东西。
